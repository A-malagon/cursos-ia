Técnicas ligeras (nivel manager)
¿Qué entiendes por MLOps?
"Es la aplicación de prácticas DevOps al ciclo de vida de los modelos ML. No solo desplegar el modelo, sino versionar los datos y el código, automatizar el reentrenamiento, monitorizar el comportamiento en producción. La diferencia clave con DevOps clásico es que en ML gestionas tres artefactos: código, datos y modelo. Y el modelo puede degradarse aunque el código no cambie, por data drift."
¿Qué es el data drift y por qué importa?
"Es cuando los datos que llegan al modelo en producción empiezan a diferir estadísticamente de los datos con los que fue entrenado. El modelo no falla técnicamente — devuelve respuestas — pero esas respuestas son cada vez menos fiables. En banca o salud eso puede ser crítico sin que nadie lo detecte. Por eso la monitorización del modelo en producción es tan importante como la de la infraestructura."

6. Entrevista Técnica — Jefe de IA

MLOps y Pipelines
¿Cómo diseñarías un pipeline ML end-to-end desde cero?
"Lo divido en cinco fases. Primero ingesta y validación de datos con Great Expectations para garantizar que los datos cumplen schema y distribuciones antes de entrenar. Segundo, feature engineering versionado con DVC para reproducibilidad. Tercero, entrenamiento con tracking en MLflow — cada experimento registra hiperparámetros, métricas y el artefacto del modelo. Cuarto, evaluación con gates de calidad — el modelo solo avanza si supera umbrales definidos con negocio. Quinto, despliegue en Kubernetes con rollout gradual y rollback inmediato. Todo orquestado con Airflow o GitHub Actions según la complejidad."
¿Qué diferencia hay entre CI/CD clásico y CI/CD para ML?
"En CI/CD clásico testeas código. En ML testeas tres cosas: código con tests normales, datos con validaciones de schema y distribución, y modelo con métricas de performance contra un baseline. Si el nuevo modelo no mejora al anterior o baja del umbral mínimo, el pipeline falla y no despliega aunque el código esté perfecto. Eso cambia completamente la lógica de los gates de calidad."
¿Cómo manejarías el reentrenamiento automático de un modelo?
"Con un trigger basado en métricas. Configuro un monitor que evalúa periódicamente la calidad del modelo en producción. Cuando una métrica cae por debajo del umbral, dispara un pipeline de reentrenamiento en Airflow. Ese pipeline coge los datos más recientes, reentrena, evalúa y si mejora al modelo en prod lo despliega automáticamente. Si no mejora, alerta al equipo para revisión manual. El truco es definir bien los umbrales y tener los datos históricos bien gobernados."
¿Qué estrategias de despliegue conoces para modelos ML?
"Las mismas que en software más una específica de ML. Blue-green: dos entornos idénticos, cambias el tráfico de golpe, rollback instantáneo. Canary: introduces el nuevo modelo para un porcentaje pequeño del tráfico, monitorizas, vas subiendo gradualmente. Shadow mode: el nuevo modelo recibe tráfico real pero sus predicciones no se usan — útil para validar sin impacto en negocio. A/B testing: comparas dos modelos midiendo impacto en métricas de negocio reales, no solo técnicas."
Infraestructura y Arquitectura
¿Cómo servirías un modelo ML en producción con alta disponibilidad?
"Empaqueto el modelo como API REST con FastAPI, containerizado con Docker. Lo despliego en Kubernetes con mínimo dos réplicas en availability zones distintas. HorizontalPodAutoscaler basado en CPU o métricas custom de latencia. Delante un Ingress con rate limiting. Para casos con requisitos de latencia muy estrictos, considero model serving especializado como Triton Inference Server, que optimiza el throughput de inferencia mejor que un servidor web genérico."
¿Qué harías si el modelo tarda demasiado en responder en producción?
"Primero mido dónde está el cuello de botella: ¿inferencia del modelo, preprocesado, latencia de red, o base de datos? Con ese diagnóstico: si es el modelo, evalúo cuantización para reducir tamaño sin perder precisión, o batch inference si el caso de uso lo permite. Si es infraestructura, escalo horizontalmente o añado caching con Redis para inputs frecuentes. En LLMs específicamente, revisar si se puede usar un modelo más pequeño para la tarea puede reducir latencia y coste dramáticamente."
¿Cómo gestionarías los secretos y credenciales en un pipeline ML?
"Nunca en el código ni en variables hardcodeadas. En Kubernetes uso External Secrets Operator integrado con Azure Key Vault. El pipeline CI/CD accede mediante Managed Identity o OIDC — sin credenciales estáticas. Los modelos en producción acceden a bases de datos mediante Managed Identities. Principio: zero secrets in code, siempre inyectados en runtime desde un vault centralizado."
LLMs y Sistemas Agénticos
¿Qué es RAG y cuándo lo usarías en lugar de fine-tuning?
"RAG en lugar de enseñarle conocimiento nuevo al modelo reentrenándolo, le das contexto relevante en cada consulta recuperándolo de una base de documentos. Elijo RAG cuando el conocimiento cambia frecuentemente: documentos de producto, normativa, datos de clientes — porque actualizar un vectorstore es trivial comparado con reentrenar. Fine-tuning lo reservo para cambiar el comportamiento o tono del modelo, no para darle conocimiento. La combinación más potente: fine-tuning para el estilo más RAG para el conocimiento."
¿Cómo funciona un sistema de agentes y cuándo lo usarías?
"Un agente es un LLM con acceso a herramientas: puede buscar en internet, consultar una base de datos, ejecutar código, llamar a una API. El ciclo es observe-think-act: observa el estado, razona qué hacer, ejecuta una acción, observa el resultado y repite. Lo uso cuando la tarea requiere múltiples pasos que no se pueden predefinir. Un sistema multi-agente añade coordinación: un orquestador delega subtareas a agentes especializados — uno busca datos, otro los analiza, otro redacta el informe."
¿Cómo monitorizas un sistema con LLMs en producción?
"En capas. Infraestructura: latencia, tokens consumidos, coste por request, tasa de error. Calidad: con LLM-as-judge, uso otro modelo para evaluar si las respuestas son relevantes y seguras en una muestra del tráfico. Seguridad: detecto prompt injection y outputs que violan guardrails. Para la trazabilidad uso LangSmith, que me permite ver exactamente qué pasó en cada cadena de razonamiento de una interacción fallida."
¿Qué es el prompt injection y cómo lo mitigas?
"Es cuando un usuario malicioso inyecta instrucciones en su input para que el LLM ignore el system prompt. Las mitigaciones son en capas: validación del input antes de llegar al LLM, system prompt que instruye explícitamente al modelo a ignorar instrucciones del usuario que contradigan su rol, guardrails de output que filtran respuestas peligrosas, y nunca dar al agente permisos que no necesite — principio de least privilege aplicado a agentes."
Criterio y Arquitectura
Si un data scientist te entrega un notebook de Jupyter, ¿qué pasos sigues para llevarlo a producción?
"Primero entiendo el modelo: inputs, outputs, dependencias, requisitos de latencia y volumen. Segundo refactorizo el notebook en código modular: preprocesado, entrenamiento e inferencia separados. Tercero añado logging y manejo de errores. Cuarto containerizo con Dockerfile. Quinto integro MLflow. Sexto construyo el pipeline CI/CD con gates de calidad. Séptimo despliegue en staging y valido con el data scientist. Octavo despliegue en producción con canary y monitorización desde el primer request."
¿Cuándo usarías Azure ML vs Azure AI Foundry?
"Azure ML para el ciclo de vida de modelos clásicos de ML: entrenamiento, registro, despliegue de modelos propios. Azure AI Foundry para todo lo relacionado con LLMs: desplegar modelos del catálogo Azure, construir sistemas con agentes, fine-tuning de modelos fundacionales. En un proyecto real usaría los dos: Azure ML para gestionar los modelos propios del cliente y AI Foundry para la capa de inteligencia conversacional encima."
¿Qué harías para reducir el coste de un sistema ML en producción?
"Análisis en tres capas. Infraestructura: revisar sizing, usar spot instances para entrenamiento, autoscaling bien configurado para no pagar capacidad idle. Modelo: cuantización, caching para inputs repetidos, batch inference donde la latencia no es crítica. En LLMs específicamente: evaluar si se puede usar un modelo más pequeño — pasar de GPT-4o a GPT-4o mini para tareas que no lo requieren reduce el coste de tokens entre 10x y 20x sin impacto perceptible en calidad."
