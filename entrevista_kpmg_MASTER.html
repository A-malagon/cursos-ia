🏦 Santander Global Tech — Cloud Privada Híbrida

Contexto: Un banco no puede meter sus sistemas críticos en cloud pública — datos regulados, sistemas legacy, compliance. La solución: construir una cloud privada on-premise con capacidad de crecer hacia cloud híbrida.

Arquitectura montada:
  • Primera fase con OpenStack como plataforma de cloud privada IaaS — gestión de compute, networking y storage sobre servidores on-premise. RedHat CloudForms como capa de orquestación y gobierno.
  • Evolución posterior hacia VMware vSphere como hipervisor principal, con vRealize Orchestrator (vRO) y vRealize Automation (VRA) como motor de servicios y marketplace de self-service.
  • VRO: motor de workflows — automatización de cualquier operación (crear VM, asignar red, aplicar política de seguridad). VRA: portal donde el equipo del banco pedía servicios de infraestructura como si fuera una tienda online.
Servicios desarrollados sobre el marketplace VRA:
  • IaaS — Servicio de VMs: workflow en VRO que automatizaba la creación, configuración y registro de una máquina virtual completa (SO, red, storage, naming convention, registro en CMDB). El usuario rellenaba un formulario en VRA y en minutos tenía su VM lista, sin intervención manual del equipo de infraestructura.
  • CaaS — Servicio de contenedores: workflow para desplegar clusters de OpenShift (Kubernetes enterprise de Red Hat) sobre la infraestructura VMware. OpenShift se integraba con el directorio activo del banco para autenticación, con el almacenamiento del banco para persistencia y con el sistema de monitorización. Se desplegaba como un servicio más del catálogo VRA — el equipo de desarrollo pedía un cluster y lo recibía aprovisionado y listo.
  • LBaaS — Load Balancer como servicio: integración de F5 BIG-IP vía API en el workflow VRO. Cuando un equipo desplegaba una aplicación, el propio workflow creaba automáticamente el virtual server en F5 con las reglas de balanceo, los health checks y el certificado SSL correspondiente. Sin ticket manual al equipo de redes.
  • BaaS — Backup como servicio: workflow integrado con Veeam Backup que al crear una VM la añadía automáticamente a la política de backup correspondiente según su criticidad (sistemas críticos: backup cada 4h, retención 90 días; sistemas estándar: backup diario, retención 30 días). Cifrado en reposo. Alertas automáticas si el backup fallaba. Cumplimiento de RPO y RTO definidos por el Banco de España.
Cloud Híbrida — qué iba a dónde:
  • On-premise / cloud privada: core bancario, datos de clientes, sistemas regulados, bases de datos transaccionales — todo con compliance ENS y guías del Banco de España.
  • Cloud pública (Azure): cargas de analítica, entornos de desarrollo y testing, herramientas de colaboración. Sin datos sensibles. Conectadas a la cloud privada por ExpressRoute / VPN privada, con Active Directory federado para identidad unificada.
  • Governance: políticas centralizadas — cualquier recurso, sea privado o público, seguía los mismos estándares de naming, tagging, seguridad y monitorización. En cloud pública se usó Terraform para IaC, garantizando que todo lo que se creaba en Azure seguía la misma plantilla aprobada.
Migración de VMs on-premise a cloud privada:
  • Desarrollé también el servicio de migración dentro del catálogo VRA/VRO: workflow que localizaba la VM en el entorno legacy, validaba su estado, la migraba a la infraestructura VMware de la cloud privada y actualizaba su registro en la CMDB automáticamente.
  • Objetivo: eliminar deuda tecnológica y centralizar la operación en la plataforma de cloud privada con todos los servicios (backup, balanceo, monitorización) aplicados de forma automática.

🔵 HPE / Roche Pharma — Migración 36.000 VMs con GreenLake

Contexto y objetivo: Roche Pharma necesitaba renovar su infraestructura de servidores físicos, pasando de hardware HPE ProLiant Gen10 a Gen11. El nuevo hardware se desplegó bajo el modelo HPE GreenLake — infraestructura on-premise gestionada por HPE con modelo de consumo como-servicio (pay-per-use), sin que Roche tuviese que comprar y gestionar el hierro directamente.

Infraestructura de destino (Gen11):
  • HPE ProLiant Gen11 como servidores físicos (cabinas de cómputo y almacenamiento).
  • Nutanix como plataforma HCI (Hyperconverged Infrastructure) — gestiona el hardware subyacente: cómputo, red y almacenamiento en un stack integrado. Las cabinas de almacenamiento se gestionan desde Nutanix Prism.
  • VMware vSphere como hipervisor de virtualización por encima de Nutanix — las VMs corren sobre vSphere, Nutanix gestiona el hierro debajo.
  • Resultado: Roche mantiene sus VMs en VMware (sin cambiar nada de las aplicaciones) pero sobre un hardware moderno, gestionado y escalable bajo GreenLake.
El reto: más de 36.000 máquinas virtuales. Hacerlo manualmente habría llevado años y el riesgo de error en sistemas críticos de una farmacéutica era inaceptable.

Cómo se ejecutó:
  • Automatización en PowerShell: desarrollé scripts que localizaban cada VM en el entorno Gen10, validaban su estado (encendida, apagada, snapshots, dependencias), ejecutaban la migración vía vMotion hacia el cluster Gen11 y verificaban que la VM arrancaba correctamente en destino.
  • Waves / tandas de migración: las 36.000 VMs se organizaron en waves (grupos de migración) ordenadas por criticidad — primero sistemas de desarrollo y pruebas, luego sistemas de soporte, finalmente sistemas críticos de producción. Cada wave tenía su ventana de mantenimiento aprobada.
  • Tablero Kanban en Trello: cada VM era una tarjeta. Los estados eran: Pendiente → En validación → En migración → Verificada → Completada. El cliente (Roche) tenía visibilidad en tiempo real del progreso sin necesitar acceso a las herramientas técnicas.
  • ServiceNow: cada wave generaba automáticamente un Change Request en SNOW. Al completar la wave, el sistema cerraba el ticket y actualizaba el registro de cada VM en la CMDB con su nueva ubicación en Gen11.
  • Liderazgo: lideré el equipo técnico de 4 personas — yo asignaba las waves, gestionaba bloqueos (VMs con dependencias cruzadas, snapshots no eliminados, licencias atadas al hardware antiguo) y era el punto de contacto técnico con el cliente.
Resultado: migración completada sin incidencias en producción, dentro de plazo y con trazabilidad total de cada VM en SNOW.
Si Javier pregunta sobre liderazgo: "Hubo momentos de presión — waves con VMs críticas en producción. Mi enfoque fue siempre datos primero: si el script detectaba un problema en la validación previa, parábamos esa VM y la escalábamos, no forzábamos. Con el equipo, repartía responsabilidad real — cada persona era dueña de su wave, yo gestionaba dependencias y desbloqueaba. Eso generaba ownership, no dependencia."

🔵 HPE / Inditex — Iniciativa FinOps

Contexto: Inditex usaba las 3 clouds principales (Azure, AWS y GCP) sin una política de gobernanza de costes definida. No había tagging consistente, equipos enteros desconocían cuánto gastaban, y se detectaron recursos activos de empresas ya extintas — por ejemplo, se seguían pagando instancias cloud de Uterque tras el cierre de esa marca.

Lo que se hizo:
  • Workshop FinOps: diseñé e impartí un taller para los equipos técnicos y financieros de Inditex sobre el modelo FinOps — cultura colaborativa entre finanzas, negocio y tecnología para gestionar el gasto cloud de forma eficiente.
  • Tagging riguroso: se definió y aplicó una taxonomía de etiquetas obligatoria en todos los recursos: env:prod/pre/dev, owner:team-name, project:nombre-proyecto, brand:zara/massimo-dutti/..., cost-center:XXXX. Ningún recurso sin tag podía crearse en los entornos tras la implementación de políticas en Azure Policy y equivalentes en AWS/GCP.
  • Rightsizing: análisis de utilización real de cada instancia. VMs con CPU al 5% de media durante 30 días → redimensionadas o eliminadas. Se liberaron recursos significativos.
  • Alertas de presupuesto: alertas automáticas al 70%, 90% y 100% del presupuesto mensual por área de negocio. El equipo de finanzas recibía el reporte, no solo IT.
  • Detección de recursos huérfanos: el análisis de tags reveló instancias sin owner activo, entornos de marcas cerradas y proyectos finalizados que seguían corriendo y facturando.
Resultado: el equipo de Inditex interiorizó la cultura FinOps — no les dejé solo dashboards, les cambié la forma de trabajar con cloud. El ahorro identificado justificó el proyecto con creces.

MLOps — infraestructura para modelos en producción:

  • Registro de modelos: Azure Machine Learning como repositorio central — versiones de modelos, linaje de datos, endpoints de inferencia gestionados.

🔵 HPE — IA Generativa: Agentes, RAG y LLMs (Actualidad)

Contexto: Actualmente lidero iniciativas internas de IA generativa en HPE — desarrollar soluciones con LLMs para proponer a clientes, con foco en automatización de procesos y acceso inteligente a información.

Stack técnico y cómo se integra:
  • LangChain como framework de orquestación — conecta el LLM con tools externas, gestiona la memoria de conversación y orquesta el flujo de los agentes.
  • RAG (Retrieval-Augmented Generation): los documentos del cliente (manuales, normativa, contratos) se trocean en chunks, se generan embeddings con Azure OpenAI (text-embedding-ada-002) y se almacenan en un vectorstore. Cuando el usuario pregunta, se recuperan los chunks relevantes y se construye el prompt con ese contexto. El LLM responde basándose solo en esos documentos — sin alucinaciones sobre contenido que no está.
  • Vectorstores: ChromaDB para desarrollo y prototipos locales (fácil, sin infraestructura). Azure AI Search para producción en Azure — gestionado, con RBAC, integrado con Azure OpenAI y cumplimiento RGPD. La elección depende del entorno del cliente.
  • Agentes con patrón ReAct: el LLM razona qué herramienta usar (Thought), la ejecuta (Action), observa el resultado (Observation) y repite. Las tools pueden ser: búsqueda RAG sobre documentos, ejecución de código Python, consulta a APIs externas, escritura de informes.
  • Multi-agente: arquitecturas donde un agente orquestador delega en agentes especializados — uno para búsqueda, otro para análisis, otro para redacción.
  • Cuándo se usa Azure AI Foundry: cuando el cliente necesita governance empresarial — gestión del ciclo de vida del agente, evaluación de calidad de respuestas, trazabilidad completa, control de acceso por equipo. Para proyectos de cliente en Azure, es la plataforma natural. Para proyectos más experimentales o multi-cloud, se desarrolla manualmente con LangChain.
  • Seguridad: Azure Key Vault para secretos, Managed Identity para autenticación sin credenciales en código, Private Endpoints para que el tráfico no salga a internet.

¿Cuál es tu visión sobre la adopción cloud en España?
España está en un momento de aceleración real. Banca, telco y utilities llevan años con estrategias híbridas pero ahora están dando el salto definitivo. El driver ya no es solo coste, es velocidad de innovación. Lo que veo es que muchas empresas tienen la infraestructura migrada pero no han transformado realmente sus procesos ni su cultura. Ahí es donde hay más trabajo, y donde KPMG aporta más valor.
¿Cómo justificarías a un CFO la migración a cloud?
Con su idioma: CapEx vs OpEx. On-premise implica inversión inicial elevada con ciclos de renovación cada 4-5 años. Cloud convierte eso en gasto operativo predecible. Pero el argumento más potente es el time-to-market: una empresa que tarda 6 meses en desplegar infraestructura no puede competir con una que lo hace en horas. Eso tiene impacto directo en ingresos, no solo en costes.
¿Cloud-first vs Hybrid Cloud?
Depende del contexto. Cloud-first tiene sentido para empresas nuevas o con poca deuda tecnológica. Hybrid cloud es la realidad de la mayoría de grandes empresas en España, especialmente en banca y telco, donde hay regulación estricta y sistemas legacy que no se migran de un día para otro. La clave no es elegir uno u otro, sino tener una estrategia clara de qué cargas van a dónde y por qué.
¿Cómo abordarías un cliente con miedo al vendor lock-in?
Es un miedo legítimo. Mi enfoque es triple: diseñar con capas de abstracción desde el principio — Kubernetes, Terraform, contenedores. Adoptar estrategia multi-cloud para cargas críticas. Y ser honesto: algo de lock-in siempre existe, la clave es gestionarlo conscientemente, no ignorarlo.
¿Qué es CaaS y cómo lo usarías en un banco?
CaaS (Container as a Service) — el proveedor gestiona la plataforma de contenedores (orquestación, networking, almacenamiento), tú solo despliegas tus imágenes Docker.

Objetivo en banca: desplegar microservicios de forma ágil, con aislamiento entre aplicaciones, escalado automático y sin gestionar la infraestructura subyacente.

Ejemplos reales:
  • Azure Kubernetes Service (AKS) — en Santander, microservicios del portal de banca online desplegados en contenedores. Cada servicio (autenticación, pagos, consultas) en su propio pod, escalando independientemente según demanda.
  • Azure Container Instances (ACI) — para tareas puntuales o batch que necesitan un contenedor que arranca, ejecuta y desaparece. Sin pagar por un cluster permanente.
  • OpenShift (Red Hat) — la versión enterprise de Kubernetes, muy usada en banca por su capa adicional de seguridad y compliance. En Santander con RedHat CloudForms la gestión de contenedores seguía esta línea.
Componentes típicos: registro de imágenes (Azure Container Registry / ACR), orquestador (AKS/OpenShift), red virtual privada para los pods, almacenamiento persistente para stateful apps, y pipeline CI/CD que construye y publica la imagen automáticamente en cada release.
¿Qué es LBaaS y para qué sirve en un entorno bancario?
LBaaS (Load Balancer as a Service) — balanceo de carga gestionado por el proveedor cloud. Distribuye el tráfico entrante entre múltiples instancias de una aplicación para garantizar alta disponibilidad y rendimiento sin gestionar appliances físicos.

Objetivo en banca: garantizar que si una instancia falla, el tráfico se redirige automáticamente a las sanas. Sin interrupciones para el cliente final. Cumplimiento de SLAs de disponibilidad que exige el regulador (99.9%+).

Ejemplos reales:
  • Azure Load Balancer (L4) — balanceo de tráfico TCP/UDP entre VMs. En Santander, distribuir peticiones entre los servidores de aplicaciones del portal interno.
  • Azure Application Gateway (L7) — balanceo inteligente con routing por URL, terminación SSL y WAF (Web Application Firewall) integrado. Crítico en banca para proteger APIs públicas.
  • F5 BIG-IP on-premise — el appliance clásico que usaba Santander antes de cloud. LBaaS es exactamente la versión cloud de esto — misma funcionalidad, sin el hierro físico.
  • Azure Traffic Manager — balanceo a nivel DNS entre regiones. Para disaster recovery: si la región primaria cae, el tráfico va automáticamente a la secundaria.
Componentes típicos: reglas de balanceo (round-robin, least connections, IP hash), health checks para detectar instancias caídas, certificados SSL gestionados, WAF para filtrar tráfico malicioso, y logs de acceso para auditoría regulatoria.
¿Qué es BaaS (Backup as a Service) y cómo se aplica en banca?
BaaS (Backup as a Service) — el proveedor cloud gestiona toda la infraestructura de backup: programación, retención, cifrado, almacenamiento y restauración. Tú defines la política, ellos ejecutan.

Objetivo en banca: cumplir con los requisitos de RPO (Recovery Point Objective — cuántos datos puedo perder) y RTO (Recovery Time Objective — en cuánto tiempo recupero) que exige el Banco de España y DORA. Un banco no puede perder más de X horas de transacciones, y debe poder recuperar sistemas en un plazo definido.

Ejemplos reales:
  • Azure Backup — en Santander, backup automatizado de VMs, bases de datos SQL Server y Azure SQL. Retención configurable (diario 30 días, semanal 12 semanas, mensual 12 meses). Cifrado en reposo con claves propias del banco en Azure Key Vault.
  • Azure Site Recovery (ASR) — va un paso más allá: no solo hace backup sino que replica las VMs en tiempo cuasi-real a otra región. Failover en minutos. En banca es la pieza clave para cumplir DORA en resiliencia operativa.
  • Veeam Backup for Azure — solución de terceros muy usada en entornos enterprise para gestión centralizada de backups multi-cloud y reporting de cumplimiento.
Componentes típicos: políticas de retención por tipo de dato (transacciones vs. logs vs. configuración), cifrado end-to-end, almacenamiento inmutable (no se puede borrar ni modificar — protección anti-ransomware), alertas de fallos de backup, y tests periódicos de restauración — obligatorios en banca para demostrar que el backup funciona realmente.
Frase para la entrevista: "En Santander la pirámide era clara — BaaS para garantizar recuperación ante desastre, LBaaS para garantizar disponibilidad ante fallos puntuales, y CaaS para garantizar que las aplicaciones se despliegan y escalan de forma ágil y controlada. Son tres capas de resiliencia complementarias."

¿Cómo diseñarías la arquitectura para un chatbot RAG sobre documentación interna?
Arquitectura en Azure en 4 capas:

Frontend: Azure Static Web Apps o App Service para la interfaz.

Orquestación: Azure Functions o App Service (Python/FastAPI) con LangChain o Semantic Kernel. Aquí vive la lógica RAG: recibe la pregunta, llama al vectorstore, construye el prompt, llama al LLM.

Datos: Azure AI Search como vectorstore. Azure Blob Storage para los documentos originales. Azure OpenAI para embeddings y respuestas (GPT-4o).

Seguridad y monitoring: Azure Key Vault para secretos. Managed Identity para autenticación sin credenciales en código. Azure Monitor + Application Insights para trazabilidad. Conditional Access para acceso solo desde la organización.
¿Qué diferencia hay entre Cloud Híbrida y Multi-cloud?
Cloud Híbrida: cloud privada + cloud pública conectadas entre sí. La clave es la conexión real entre ambos entornos — red, identidad y seguridad compartidas. Ejemplo: Santander tenía cloud privada VMware + Azure conectadas por ExpressRoute, con Active Directory federado para identidad unificada.

Multi-cloud: dos o más clouds públicas a la vez. No implica cloud privada. Ejemplo: Inditex usaba Azure + AWS + GCP simultáneamente — eso es multi-cloud.

On-premise / Legacy: servidores físicos propios sin virtualización cloud. Coexisten con la arquitectura cloud pero no forman parte de ella. En Santander había sistemas legacy on-premise que no se migraron porque el coste y el riesgo no lo justificaban.

Pueden combinarse: en banca lo habitual es tener los tres a la vez — legacy on-premise + cloud privada + Azure como cloud pública. En la práctica se llama cloud híbrida aunque también haya sistemas legacy que no tocan ninguna cloud.
¿Cómo se enlazan dos clouds públicas en multi-cloud?
Las clouds públicas no se conectan directamente entre sí de forma nativa. Tres opciones principales:

1. VPN Site-to-Site: túnel IPSec cifrado entre Azure VNet y AWS VPC. Fácil y barato, pero la latencia es variable porque va por internet público. Válido para cargas no críticas.

2. Interconexión privada dedicada: ExpressRoute (Azure) + Direct Connect (AWS) terminando en un punto neutro como Equinix. El tráfico nunca va por internet — latencia baja y predecible. Lo que usa banca con datos sensibles en multi-cloud.

3. SD-WAN / Red overlay: una capa de red virtual (Cisco SD-WAN, VMware NSX) por encima de todas las clouds que gestiona el tráfico de forma centralizada.

Qué se comparte en multi-cloud: identidad federada entre Azure AD y AWS IAM vía SAML/OIDC, políticas de seguridad centralizadas (Microsoft Defender multi-cloud, Prisma Cloud) y herramientas FinOps unificadas para ver el gasto de las 3 clouds en un solo sitio — exactamente lo que implementé en Inditex con Azure, AWS y GCP.
¿Cómo se enlaza una cloud privada con una cloud pública en cloud híbrida?
Opción 1 — VPN Site-to-Site: túnel IPSec cifrado entre la cloud privada y Azure VNet. Rápido de montar y barato, pero la latencia depende del ISP porque va por internet.

Opción 2 — ExpressRoute (lo que usa banca): circuito físico dedicado entre el datacenter y la región de Azure — sin pasar por internet público. Latencia fija, baja y predecible, ancho de banda garantizado (1Gbps, 10Gbps). En Santander era la opción obligatoria — datos regulados no pueden ir por internet público. Era el enlace entre la cloud privada VMware y Azure.

Qué se comparte en cloud híbrida:
  • Identidad: Active Directory on-premise federado con Azure AD — mismo usuario, misma contraseña en ambos entornos
  • Red: rangos de IPs diferentes en cada entorno, enrutados entre sí por el ExpressRoute o VPN
  • DNS: DNS privado que resuelve nombres tanto de recursos on-prem como de Azure
  • Seguridad: políticas aplicadas en ambos lados — en Santander con VMware NSX para microsegmentación en la parte privada
  • IaC: Terraform para garantizar que todo lo creado en Azure seguía la misma plantilla aprobada
¿Qué diferencia hay entre IaaS, PaaS y SaaS? Pon ejemplos reales.
IaaS (Infrastructure as a Service) — el proveedor pone el hardware, tú gestionas el SO, middleware y apps. Ejemplos reales en banca: Azure Virtual Machines para servidores de aplicaciones internas, Azure Virtual Network para segmentar entornos prod/pre/dev, Azure Blob Storage para logs y backups, Nutanix/VMware on-premise como cloud privada — que es exactamente lo que monté en Santander con OpenStack.

PaaS (Platform as a Service) — tú solo traes el código, el proveedor gestiona SO, red, escalado y parcheo. Ejemplos: Azure Kubernetes Service para microservicios sin gestionar nodos, Azure SQL Database sin DBA que parchee servidores, Azure Machine Learning para entrenar modelos de riesgo o fraude, Azure DevOps para CI/CD de aplicaciones bancarias, Azure OpenAI Service — LLMs gestionados sin tocar GPUs.

SaaS (Software as a Service) — usas la aplicación directamente, no gestionas nada. Ejemplos que usa Santander: Microsoft 365 (Teams, Outlook) para comunicación corporativa, Salesforce como CRM, ServiceNow para gestión de incidencias IT, Power BI para reporting regulatorio.

La frontera en banca es clara: IaaS para lo que necesita control total, PaaS para acelerar desarrollo sin gestionar infraestructura, y SaaS para herramientas corporativas estándar donde no hay diferenciación competitiva en operarlas tú mismo.
¿Terraform es IaaS? ¿Cómo lo clasificarías?
Terraform no es IaaS — es IaC (Infrastructure as Code), una herramienta que aprovisiona recursos IaaS. La distinción es importante. Terraform es el arquitecto que dibuja los planos; Azure o AWS es la constructora que levanta el edificio; la VM o la red resultante es el IaaS. Lo que hace Terraform es describir en código qué infraestructura quieres y el proveedor la crea. Lo usé en HPE e Inditex para aprovisionar infraestructura cloud de forma reproducible y versionada — cualquier cambio pasa por Git y por el pipeline de CI/CD antes de tocar producción. En KPMG eso se enmarcaría dentro de cloud governance e Infrastructure as Code, que es una práctica transversal que se aplica sobre IaaS.
¿Cómo escalarías un modelo de ML en producción?
Depende del patrón de carga. Si es batch: Azure ML Pipelines con compute clusters que escalan a cero cuando no se usan. Si es online (API en tiempo real): AKS con Horizontal Pod Autoscaler — escala según CPU/memoria o métricas custom. Si hay picos muy variables: Azure ML Managed Online Endpoints con autoscaling. Para un consultor, la pregunta clave al cliente es: ¿cuántas predicciones por segundo en pico? ¿Latencia máxima aceptable? ¿Los picos son predecibles?
¿Qué diferencia una operación IT tradicional de una operación cloud-native?
En IT tradicional, la infraestructura se gestiona de forma reactiva: falla un servidor y se interviene manualmente. En cloud-native el modelo es proactivo e inmutable: no se reparan servidores, se reemplazan instancias. La observabilidad pasa de monitorizar hardware a monitorizar métricas de aplicación. Los equipos de operaciones pasan de ser guardianes de los servidores a ingenieros de plataforma que gestionan código — IaC, pipelines, configuración declarativa. El resultado es menor MTTR, más consistencia y operación a escala sin escalar el equipo linealmente.
¿Cómo gestionarías la resistencia de un equipo de operaciones tradicional ante la adopción cloud?
Con empatía y victorias rápidas, no con mandato. La resistencia suele venir del miedo: miedo a perder el trabajo, miedo a no saber. Lo que funciona es: primero involucrarles en el diseño, no imponerles la solución. Segundo, mostrar que sus skills son el fundamento — quien sabe operar servidores tiene ventaja enorme para operar Kubernetes. Tercero, pequeños pilotos donde ellos sean los protagonistas del éxito. Javier González lleva 15 años en Operaciones — sabe perfectamente que la resistencia interna es el mayor bloqueador de transformación.

Conceptos IA — LLMs, Embeddings, Fundamentos

LLMs — Large Language Models

¿Qué es un LLM y cómo funciona a grandes rasgos?
Un LLM es un modelo de lenguaje entrenado con enormes cantidades de texto para predecir qué token viene a continuación. Durante el entrenamiento aprende patrones estadísticos del lenguaje humano. En inferencia, dada una entrada (prompt), genera una respuesta token a token. Lo que parece "razonamiento" es en realidad interpolación sofisticada sobre patrones aprendidos. Los modelos actuales como GPT-4o o Claude tienen ventanas de contexto de 128K-200K tokens, lo que permite conversaciones largas y análisis de documentos completos.
¿Qué es un token?
La unidad mínima de procesamiento del LLM. No es una palabra — es un fragmento de texto. "MLOps" puede ser 1 o 3 tokens según el modelo. Importante para un consultor porque el coste de la API se mide en tokens: coste = (tokens entrada + tokens salida) × precio por token. Para un cliente con millones de llamadas, la gestión de tokens puede suponer diferencias de decenas de miles de euros al mes.
¿Qué es la temperatura en un LLM?
Parámetro que controla la aleatoriedad de las respuestas. Temperatura 0 → siempre la respuesta más probable (producción, código, respuestas factuales). Temperatura 0.7 → balance creatividad/coherencia (chatbots de atención). Temperatura 1+ → muy creativo, puede ser incoherente (brainstorming). Para un cliente bancario que usa IA para responder preguntas regulatorias, temperatura 0 o muy baja es obligatoria.

Embeddings

¿Qué son los embeddings y para qué sirven?
Un embedding es la representación numérica de un texto como vector en un espacio de alta dimensión (1536 números con text-embedding-ada-002). Lo relevante: textos con significado similar tienen vectores cercanos. "impago" y "riesgo de crédito" estarán cerca. "impago" y "receta de cocina" estarán lejos. Se usan en tres casos principales: RAG (encontrar documentos relevantes para una pregunta), búsqueda semántica (búsqueda por significado, no por palabras clave), y clasificación de textos (agrupar tickets de soporte por tema).
¿Qué diferencia hay entre búsqueda semántica y búsqueda por palabras clave?
Búsqueda por palabras clave (Elasticsearch clásico): busca la palabra exacta. Si el documento dice "impago" y la consulta es "cliente que no paga", no lo encuentra. Búsqueda semántica con embeddings: compara el significado. "cliente que no paga" y "riesgo de impago" son semánticamente similares → sí los conecta. Para un banco que quiere que sus analistas busquen en normativa regulatoria o contratos, la búsqueda semántica es transformacional.

Fine-tuning vs RAG

¿Cuándo recomiendas fine-tuning y cuándo RAG?
Fine-tuning: modificas los pesos del modelo con tus propios datos. Enseñas al modelo a hablar de una manera concreta, seguir un formato específico o conocer terminología de nicho. Caro (GPUs, tiempo, datos etiquetados) y difícil de actualizar — si cambia la información, tienes que reentrenar. RAG: el modelo base no cambia. Le das contexto relevante en el prompt en tiempo real. Mucho más barato, actualizable al instante y auditable. Para la mayoría de casos empresariales (documentos que cambian, normativa, bases de conocimiento) RAG es la respuesta correcta. Fine-tuning tiene sentido si necesitas cambiar el estilo/formato de respuesta o si la terminología es muy especializada y el modelo base la desconoce completamente.
Regla para la entrevista: si te preguntan "fine-tuning o RAG", di RAG salvo que haya una razón muy específica. Es la respuesta que da un consultor experimentado, no "depende" sin más.

MLOps

¿Qué es MLOps y por qué lo necesita una empresa?
MLOps es la aplicación de prácticas DevOps al ciclo de vida de modelos de Machine Learning. Un modelo no es código estático — su rendimiento degrada con el tiempo porque los datos del mundo real cambian (data drift) o el comportamiento que modela cambia (concept drift). MLOps incluye: versionar datos (DVC) y modelos (MLflow), automatizar el reentrenamiento cuando la calidad cae, tener pipelines CI/CD para modelos, monitorizar el modelo en producción (Evidently, Azure Monitor). Sin MLOps, los modelos se degradan en silencio y nadie lo sabe hasta que el negocio se queja. Lo he aplicado en HPE desplegando modelos con Flask y Model Serving en cloud.
¿Qué diferencia hay entre MLflow y Airflow?
MLflow: tracking de experimentos de ML. Registra métricas, parámetros, artefactos (modelos, gráficas) de cada entrenamiento. Permite comparar runs y registrar el modelo que va a producción. Responde a "¿qué modelo entrené, con qué datos, y qué métricas obtuvo?". Airflow: orquestador de workflows. Define pipelines de datos/ML como DAGs con dependencias, scheduling y reintentos. Responde a "¿cuándo y en qué orden ejecuto cada paso de mi pipeline?". Se usan juntos: Airflow orquesta el pipeline completo, MLflow trackea lo que ocurre durante el entrenamiento. Son complementarios, no competidores.
¿Cómo ves el impacto de la IA en infraestructura cloud?
De formas muy concretas. AIOps para detectar anomalías antes de que causen incidencias. Optimización de costes automática con rightsizing inteligente. Y generación de código IaC con LLMs, que acelera el trabajo de los equipos de plataforma. La IA no reemplaza al cloud engineer, pero multiplica su productividad. RAG en profundidad

RAG (Retrieval-Augmented Generation) es el patrón de arquitectura más usado en proyectos de IA empresarial. Debes poder explicarlo de memoria y dibujarlo si te lo piden.

Flujo completo RAG

[FASE INDEXACIÓN — se hace una vez] Documentos (PDFs, Word, contratos, normativa...) ↓ Dividir en chunks (trozos de ~300-500 tokens) ↓ Generar embedding de cada chunk (Azure OpenAI: text-embedding-ada-002) ↓ Guardar embeddings + texto en Vectorstore (ChromaDB / FAISS / Azure AI Search) [FASE CONSULTA — en tiempo real] Usuario hace una pregunta ↓ Generar embedding de la pregunta ↓ Buscar los N chunks más similares en el vectorstore (búsqueda semántica) ↓ Construir prompt: "Pregunta: X | Contexto relevante: chunk1, chunk2, chunk3" ↓ Enviar al LLM (Azure OpenAI GPT-4o) ↓ LLM responde basándose SOLO en el contexto proporcionado
¿Por qué RAG en vez de meter todos los documentos en el contexto del LLM?
Por coste y latencia. Un contrato bancario puede ser 100 páginas. Meter 100 páginas en cada pregunta costaría una fortuna en tokens y haría cada llamada lenta. RAG selecciona solo los 3-5 fragmentos más relevantes para cada pregunta. Además, los LLMs con contextos muy largos tienen peor rendimiento en los fragmentos del "medio" del contexto (lost-in-the-middle problem). RAG mantiene el contexto pequeño y relevante.
¿Qué es el chunking y por qué importa?
Chunking es cómo divides los documentos antes de indexarlos. Si el chunk es demasiado pequeño, pierde contexto. Si es demasiado grande, el embedding es menos preciso y el retrieval peor. El tamaño típico es 300-1000 tokens con overlap (solapamiento) del 10-20% para no perder contexto en los cortes. Para documentos estructurados como normativa o contratos, es mejor hacer chunking por sección lógica que por número fijo de tokens.
¿Qué vectorstore recomiendas y cuándo?
Cuatro opciones principales: ChromaDB para desarrollo y prototipos. FAISS para velocidad máxima en memoria. Milvus para producción a escala con millones de vectores. Para un cliente Azure empresarial: Azure AI Search — el vectorstore gestionado de Microsoft, integrado nativamente con Azure OpenAI, con RBAC, cumplimiento normativo y sin gestionar infraestructura adicional.

RAG avanzado

¿Qué problemas tiene RAG y cómo se mitigan?
Tres problemas principales: (1) Retrieval incorrecto — si los chunks recuperados no son los adecuados, el LLM genera una respuesta basada en contexto incorrecto. Mitigación: re-ranking, HyDE (generar una respuesta hipotética y buscar chunks similares). (2) Alucinaciones — el LLM puede generar información que no está en los chunks. Mitigación: instrucciones explícitas de "responde solo con la información del contexto". (3) Contexto fragmentado. Mitigación: overlap entre chunks, parent-child chunking.
¿Cómo medirías el éxito de un proyecto RAG en producción?
Cuatro métricas: (1) Relevancia del retrieval — ¿los chunks recuperados son los correctos? (2) Calidad de la respuesta — ¿correcta y fundamentada en los chunks? RAGAS es un framework específico para esto. (3) Tasa de "no sé" — preferimos alto "no sé" a alucinaciones. (4) Satisfacción del usuario — thumbs up/down en la UI.

Agentes IA

Los agentes son LLMs que pueden tomar decisiones, usar herramientas externas y ejecutar secuencias de pasos para completar una tarea.

¿Qué es un agente de IA y en qué se diferencia de un chatbot normal?
Un chatbot normal: recibes un mensaje → generas una respuesta → fin. Un agente: recibes una tarea → el LLM decide qué herramientas usar → ejecuta acciones → observa el resultado → decide el siguiente paso → repite hasta completar la tarea. La diferencia clave es la capacidad de acción: un agente puede buscar en Google, ejecutar código, consultar una base de datos, enviar un email o llamar a una API. No solo genera texto — genera texto Y actúa.
¿Qué es el patrón ReAct?
ReAct (Reasoning + Acting) es el patrón fundamental de los agentes modernos. El LLM alterna entre: Thought (razonamiento interno — "necesito saber el precio actual de IBEX para calcular esto"), Action (usa una herramienta), Observation (observa el resultado), y vuelve a Thought. Este ciclo se repite hasta llegar a la respuesta final. Es el patrón que usan LangChain Agents, AutoGPT y la mayoría de frameworks de agentes.
¿Qué riesgos tienen los agentes en producción empresarial?
Cuatro riesgos que un consultor debe conocer: (1) Prompt injection — un usuario malicioso puede intentar que el agente ignore sus instrucciones. (2) Scope creep — el agente puede encadenar acciones que van más allá de lo que el usuario pretendía. (3) Costes impredecibles — un agente que entra en bucle puede hacer decenas de llamadas API. (4) Trazabilidad y auditoría — en sectores regulados es obligatorio poder explicar por qué el agente tomó cada decisión. Soluciones: guardrails estrictos, límite de pasos, aprobación humana para acciones críticas (human-in-the-loop), logging completo.
¿Qué frameworks de agentes conoces?
LangChain es el más extendido. LangGraph (de los mismos creadores) añade control de flujo explícito con grafos de estado, más adecuado para agentes complejos en producción. Semantic Kernel de Microsoft es el más integrado con el ecosistema Azure — plugins, planners, memoria semántica. AutoGen de Microsoft Research para sistemas multi-agente. Para un cliente KPMG en Azure, Semantic Kernel + Azure AI Foundry es la combinación natural.

Caso de uso concreto para KPMG: un agente que ayuda a analistas de auditoría a revisar contratos. El orquestador recibe "analiza este contrato y busca cláusulas de riesgo". El agente de búsqueda recupera cláusulas similares de contratos pasados. El agente de análisis las compara. El agente de validación verifica que el informe cumple la plantilla estándar.
¿Cómo garantizarías la trazabilidad de un agente en un entorno regulado como banca?
Es obligatorio y no negociable. El enfoque es: logging estructurado de cada ciclo Thought-Action-Observation con timestamp, usuario y contexto. Almacenamiento en Azure Monitor o Cosmos DB con retención conforme a normativa. Uso de LangSmith o Azure AI Foundry's tracing para visualizar el grafo de decisiones completo. Y human-in-the-loop para acciones que impliquen datos sensibles o decisiones de alto impacto — el agente propone, el humano aprueba. La GDPR y DORA requieren poder explicar cualquier decisión automatizada que afecte a un cliente.

AIOps & Operaciones IT — Especialidad de Javier

Javier González lleva +15 años en Infraestructura, Arquitectura y Operación IT y lidera iniciativas de AIOps y DevOps. Este es su terreno. Aquí puedes diferenciarte.

¿Qué es AIOps y qué problema resuelve?
AIOps es la aplicación de IA y ML a las operaciones IT para pasar de una gestión reactiva a una proactiva. El problema que resuelve es el volumen de ruido en operaciones modernas: miles de alertas al día de las que solo el 5% son relevantes. AIOps permite: correlación automática de alertas para detectar causas raíz, predicción de incidencias antes de que impacten, análisis de logs con NLP para detectar patrones anómalos, y routing inteligente de incidencias. El resultado: menos MTTR, menos noches sin dormir y equipos de operaciones que se centran en lo que importa.
¿Cómo se implementa AIOps en la práctica para un banco?
En fases. Primero la base: observabilidad unificada — todos los logs, métricas y trazas centralizados (Azure Monitor, Elastic, Datadog). Sin eso, no hay AIOps real. Segunda fase: correlación inteligente de alertas — eliminar el ruido, agrupar alertas relacionadas en un único incidente. Tercera fase: detección de anomalías con ML sobre series temporales — detectar que el tiempo de respuesta de un servicio crítico está degradándose antes de que supere el umbral. Cuarta fase: automatización de remediación — si el modelo tiene alta confianza y el impacto es bajo, que el propio sistema actúe. Lo más avanzado, y donde hay más cautela en banca por regulación.
¿Qué herramientas de AIOps conoces?
En el ecosistema Microsoft: Azure Monitor con machine learning integrado, Azure Sentinel para el ámbito de seguridad (SIEM con IA). Plataformas especializadas: Dynatrace (el más avanzado en IA para observabilidad, Davis es su motor IA), Datadog (muy fuerte en cloud-native), Splunk (más orientado a seguridad y logs). Para la parte de ITSM con IA: ServiceNow AIOps integra con estas plataformas para automatizar la gestión de incidencias. En un proyecto KPMG con un banco que ya tenga Azure, el stack natural es Azure Monitor + Dynatrace o Datadog + ServiceNow.
¿Cómo justificarías la inversión en AIOps a un comité directivo de un banco?
Con dos números: MTTR y disponibilidad. El MTTR (Mean Time To Resolve) en operaciones IT tradicionales de un banco grande puede ser 2-4 horas por incidencia mayor. AIOps puede reducirlo un 40-60% en incidencias con patrones conocidos. Y cada minuto de indisponibilidad en banca online puede costar entre 50.000 y 500.000€ según el banco. El ROI se calcula fácil: cuántas incidencias mayores al año, cuánto cuestan por minuto, cuánto reduce el MTTR AIOps. Más un argumento regulatorio: DORA exige demostrar resiliencia operativa — AIOps es evidencia de madurez operativa ante el regulador.
¿Cómo conectarías tu experiencia en automatización con lo que necesita KPMG en operaciones?
La migración de 36.000 VMs en Roche es el ejemplo más directo: un proyecto de operaciones IT a escala enorme, donde la única forma de ejecutarlo sin incidencias fue automatización total con validación en cada paso. Eso es exactamente lo que AIOps busca — operaciones que se validan, monitorizan y en última instancia ejecutan solas. Mi experiencia en Santander con aprovisionamiento automatizado con Ansible y monitorización con Grafana es la misma filosofía: operar infraestructura compleja con código, no con intervención manual. Es el puente natural entre lo que he hecho y lo que lidera Javier en KPMG.
¿Qué es DevOps y cómo lo diferencias de SRE?
DevOps es una cultura y conjunto de prácticas que une desarrollo y operaciones para acelerar el ciclo de entrega de software. Se centra en CI/CD, automatización, feedback continuo. SRE (Site Reliability Engineering, de Google) es una implementación concreta de DevOps con foco en fiabilidad: define SLOs (Service Level Objectives), gestiona error budgets y automatiza la operación para que los ingenieros puedan centrarse en reducir el toil manual. En la práctica: DevOps es el "qué" y SRE es el "cómo" operativo. Para un banco, SRE aporta el rigor cuantitativo que el regulador aprecia.

Preguntas Trampa — Las que más penalizan

Estas son respuestas que suenan bien pero que un entrevistador técnico de KPMG sabrá que son incorrectas o superficiales.

Trampa 1: "Los LLMs razonan"

Error: decir que los LLMs "razonan", "entienden" o "piensan".

Correcto: Los LLMs predicen el siguiente token basándose en patrones estadísticos. Lo que parece razonamiento es interpolación sofisticada. No son fiables para lógica formal o matemáticas exactas, pueden contradecirse.

Trampa 2: "RAG elimina las alucinaciones"

Error: decir que RAG resuelve el problema de alucinaciones.

Correcto: RAG las reduce significativamente al anclar las respuestas en documentos reales. Pero si el retrieval falla (devuelve chunks irrelevantes), el LLM puede alucinar igualmente. RAG reduce el riesgo, no lo elimina.

Trampa 3: "Fine-tuning siempre da mejores resultados"

Error: asumir que fine-tuning > RAG en calidad.

Correcto: Para conocimiento factual actualizable, RAG supera al fine-tuning. El fine-tuning enseña estilo y formato, no hechos. Un modelo fine-tuneado con documentos de enero puede alucinar hechos de marzo.

Trampa 4: "Más contexto siempre es mejor"

Error: creer que meter más contexto en el prompt mejora las respuestas.

Correcto: Los LLMs tienen el problema "lost in the middle" — dan menos importancia a información en el centro de contextos muy largos. Contextos muy largos = más latencia y más coste.

Trampa 5: Confundir Airflow con MLflow

Error: usar estos términos indistintamente o confundir sus funciones.

Correcto: Airflow = orquestación de pipelines (cuándo y en qué orden). MLflow = tracking de experimentos y registro de modelos. Son complementarios, no competidores.

Trampa 6: Vectorstore = base de datos relacional con vectores

Error: describir un vectorstore como "una base de datos que guarda vectores".

Correcto: Un vectorstore es una base de datos especializada en búsqueda de similitud por distancia (coseno, euclidiana). El motor de búsqueda es un índice ANN (Approximate Nearest Neighbor) que permite buscar los K vectores más cercanos entre millones en milisegundos.

Trampa 7 (nueva): "AIOps es solo monitorización avanzada"

Error: reducir AIOps a "monitorización con dashboards".

Correcto: AIOps va más allá de observar — incluye correlación automática de alertas, predicción de fallos, análisis de causa raíz y remediación automatizada. La diferencia es la capacidad de acción proactiva, no solo la visualización.
