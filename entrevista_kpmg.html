Cloud & Arquitectura

¿Cuál es tu visión sobre la adopción cloud en España?
España está en un momento de aceleración real. Banca, telco y utilities llevan años con estrategias híbridas pero ahora están dando el salto definitivo. El driver ya no es solo coste, es velocidad de innovación. Lo que veo es que muchas empresas tienen la infraestructura migrada pero no han transformado realmente sus procesos ni su cultura. Ahí es donde hay más trabajo, y donde KPMG aporta más valor.
¿Cómo justificarías a un CFO la migración a cloud?
Con su idioma: CapEx vs OpEx. On-premise implica inversión inicial elevada con ciclos de renovación cada 4-5 años. Cloud convierte eso en gasto operativo predecible. Pero el argumento más potente es el time-to-market: una empresa que tarda 6 meses en desplegar infraestructura no puede competir con una que lo hace en horas. Eso tiene impacto directo en ingresos, no solo en costes.
¿Cloud-first vs Hybrid Cloud?
Depende del contexto. Cloud-first tiene sentido para empresas nuevas o con poca deuda tecnológica. Hybrid cloud es la realidad de la mayoría de grandes empresas en España, especialmente en banca y telco, donde hay regulación estricta y sistemas legacy que no se migran de un día para otro. La clave no es elegir uno u otro, sino tener una estrategia clara de qué cargas van a dónde y por qué.
¿Cómo abordarías un cliente con miedo al vendor lock-in?
Es un miedo legítimo. Mi enfoque es triple: diseñar con capas de abstracción desde el principio — Kubernetes, Terraform, contenedores. Adoptar estrategia multi-cloud para cargas críticas. Y ser honesto: algo de lock-in siempre existe, la clave es gestionarlo conscientemente, no ignorarlo.
¿Cómo diseñarías una arquitectura multi-cloud para un banco?
Partiría de una Landing Zone en cada cloud con governance centralizado — políticas, seguridad, costes. Terraform para IaC agnóstico al proveedor, Kubernetes para las aplicaciones, identidad federada con Azure AD. Las cargas más sensibles — core bancario — se quedan on-premise o en cloud privada, como hice en Santander. Las analíticas y de innovación van a cloud pública. Y modelo FinOps desde el día uno, como apliqué en Inditex.
¿Qué es FinOps y cómo lo has aplicado?
FinOps es gestionar el gasto cloud de forma colaborativa entre finanzas, negocio y tecnología. Lo apliqué en HPE con Inditex: tagging riguroso, alertas de presupuesto, revisión periódica de rightsizing. Y diseñé e impartí un workshop para que el equipo de Inditex lo interiorizara. El objetivo es optimizar sin frenar la innovación.
¿Qué es una Landing Zone?
La base estructurada sobre la que construyes todo en cloud — una plantilla de suscripción/cuenta bien configurada con governance, seguridad, networking e identidad desde el principio. En Azure el framework de referencia es el Cloud Adoption Framework. En vez de que cada equipo cree suscripciones como quiera, la Landing Zone garantiza que todo parte de los mismos estándares.
¿Cómo garantizas seguridad y compliance en cloud para banca?
En banca hay que cumplir ENS, DORA y guías del Banco de España. Mi enfoque es security by design: IAM con mínimo privilegio, cifrado en tránsito y en reposo, segmentación de red, auditoría continua. Y compliance as code — las políticas se definen en código y se validan en el pipeline CI/CD antes de que nada llegue a producción. Lo implementé en Santander donde el escrutinio regulatorio era máximo.
¿Cómo diseñarías la arquitectura para un chatbot RAG sobre documentación interna?
Arquitectura en Azure en 4 capas:

Frontend: Azure Static Web Apps o App Service para la interfaz.

Orquestación: Azure Functions o App Service (Python/FastAPI) con LangChain o Semantic Kernel. Aquí vive la lógica RAG: recibe la pregunta, llama al vectorstore, construye el prompt, llama al LLM.

Datos: Azure AI Search como vectorstore. Azure Blob Storage para los documentos originales. Azure OpenAI para embeddings y respuestas (GPT-4o).

Seguridad y monitoring: Azure Key Vault para secretos. Managed Identity para autenticación sin credenciales en código. Azure Monitor + Application Insights para trazabilidad. Conditional Access para acceso solo desde la organización.
¿Cómo escalarías un modelo de ML en producción?
Depende del patrón de carga. Si es batch: Azure ML Pipelines con compute clusters que escalan a cero cuando no se usan. Si es online (API en tiempo real): AKS con Horizontal Pod Autoscaler — escala según CPU/memoria o métricas custom. Si hay picos muy variables: Azure ML Managed Online Endpoints con autoscaling. Para un consultor, la pregunta clave al cliente es: ¿cuántas predicciones por segundo en pico? ¿Latencia máxima aceptable? ¿Los picos son predecibles?

Conceptos IA — LLMs, Embeddings, Fundamentos

LLMs — Large Language Models

¿Qué es un LLM y cómo funciona a grandes rasgos?
Un LLM es un modelo de lenguaje entrenado con enormes cantidades de texto para predecir qué token viene a continuación. Durante el entrenamiento aprende patrones estadísticos del lenguaje humano. En inferencia, dada una entrada (prompt), genera una respuesta token a token. Lo que parece "razonamiento" es en realidad interpolación sofisticada sobre patrones aprendidos. Los modelos actuales como GPT-4o o Claude tienen ventanas de contexto de 128K-200K tokens, lo que permite conversaciones largas y análisis de documentos completos.
¿Qué es un token?
La unidad mínima de procesamiento del LLM. No es una palabra — es un fragmento de texto. "MLOps" puede ser 1 o 3 tokens según el modelo. Importante para un consultor porque el coste de la API se mide en tokens: coste = (tokens entrada + tokens salida) × precio por token. Para un cliente con millones de llamadas, la gestión de tokens puede suponer diferencias de decenas de miles de euros al mes.
¿Qué es la temperatura en un LLM?
Parámetro que controla la aleatoriedad de las respuestas. Temperatura 0 → siempre la respuesta más probable (producción, código, respuestas factuales). Temperatura 0.7 → balance creatividad/coherencia (chatbots de atención). Temperatura 1+ → muy creativo, puede ser incoherente (brainstorming). Para un cliente bancario que usa IA para responder preguntas regulatorias, temperatura 0 o muy baja es obligatoria.

Embeddings

¿Qué son los embeddings y para qué sirven?
Un embedding es la representación numérica de un texto como vector en un espacio de alta dimensión (1536 números con text-embedding-ada-002). Lo relevante: textos con significado similar tienen vectores cercanos. "impago" y "riesgo de crédito" estarán cerca. "impago" y "receta de cocina" estarán lejos. Se usan en tres casos principales: RAG (encontrar documentos relevantes para una pregunta), búsqueda semántica (búsqueda por significado, no por palabras clave), y clasificación de textos (agrupar tickets de soporte por tema).
¿Qué diferencia hay entre búsqueda semántica y búsqueda por palabras clave?
Búsqueda por palabras clave (Elasticsearch clásico): busca la palabra exacta. Si el documento dice "impago" y la consulta es "cliente que no paga", no lo encuentra. Búsqueda semántica con embeddings: compara el significado. "cliente que no paga" y "riesgo de impago" son semánticamente similares → sí los conecta. Para un banco que quiere que sus analistas busquen en normativa regulatoria o contratos, la búsqueda semántica es transformacional.

Fine-tuning vs RAG — pregunta clásica de consultor

¿Cuándo recomiendas fine-tuning y cuándo RAG?
Fine-tuning: modificas los pesos del modelo con tus propios datos. Enseñas al modelo a hablar de una manera concreta, seguir un formato específico o conocer terminología de nicho. Caro (GPUs, tiempo, datos etiquetados) y difícil de actualizar — si cambia la información, tienes que reentrenar. RAG: el modelo base no cambia. Le das contexto relevante en el prompt en tiempo real. Mucho más barato, actualizable al instante y auditable. Para la mayoría de casos empresariales (documentos que cambian, normativa, bases de conocimiento) RAG es la respuesta correcta. Fine-tuning tiene sentido si necesitas cambiar el estilo/formato de respuesta o si la terminología es muy especializada y el modelo base la desconoce completamente.
Regla para la entrevista: si te preguntan "fine-tuning o RAG", di RAG salvo que haya una razón muy específica. Es la respuesta que da un consultor experimentado, no "depende" sin más.

MLOps

¿Qué es MLOps y por qué lo necesita una empresa?
MLOps es la aplicación de prácticas DevOps al ciclo de vida de modelos de Machine Learning. Un modelo no es código estático — su rendimiento degrada con el tiempo porque los datos del mundo real cambian (data drift) o el comportamiento que modela cambia (concept drift). MLOps incluye: versionar datos (DVC) y modelos (MLflow), automatizar el reentrenamiento cuando la calidad cae, tener pipelines CI/CD para modelos, monitorizar el modelo en producción (Evidently, Azure Monitor). Sin MLOps, los modelos se degradan en silencio y nadie lo sabe hasta que el negocio se queja. Lo he aplicado en HPE desplegando modelos con Flask y Model Serving en cloud.
¿Qué diferencia hay entre MLflow y Airflow?
MLflow: tracking de experimentos de ML. Registra métricas, parámetros, artefactos (modelos, gráficas) de cada entrenamiento. Permite comparar runs y registrar el modelo que va a producción. Responde a "¿qué modelo entrené, con qué datos, y qué métricas obtuvo?". Airflow: orquestador de workflows. Define pipelines de datos/ML como DAGs (grafos acíclicos dirigidos) con dependencias, scheduling y reintentos. Responde a "¿cuándo y en qué orden ejecuto cada paso de mi pipeline?". Se usan juntos: Airflow orquesta el pipeline completo, MLflow trackea lo que ocurre durante el entrenamiento. Son complementarios, no competidores.
¿Cómo ves el impacto de la IA en infraestructura cloud?
De formas muy concretas. AIOps para detectar anomalías antes de que causen incidencias. Optimización de costes automática con rightsizing inteligente. Y generación de código IaC con LLMs, que acelera el trabajo de los equipos de plataforma. La IA no reemplaza al cloud engineer, pero multiplica su productividad. En KPMG veo una oportunidad enorme de llevar esto a clientes que todavía gestionan infraestructura de forma reactiva.
¿Cómo venderías IA a un cliente tradicional?

Luego presento la solución, que resulta que usa IA por debajo. Los clientes tradicionales compran resultados, no tecnología. Con un ROI estimado y un caso de uso concreto, la conversación cambia completamente.

RAG en profundidad

RAG (Retrieval-Augmented Generation) es el patrón de arquitectura más usado en proyectos de IA empresarial. Debes poder explicarlo de memoria y dibujarlo si te lo piden.

Flujo completo RAG

[FASE INDEXACIÓN — se hace una vez] Documentos (PDFs, Word, contratos, normativa...) ↓ Dividir en chunks (trozos de ~300-500 tokens) ↓ Generar embedding de cada chunk (Azure OpenAI: text-embedding-ada-002) ↓ Guardar embeddings + texto en Vectorstore (ChromaDB / FAISS / Azure AI Search) [FASE CONSULTA — en tiempo real] Usuario hace una pregunta ↓ Generar embedding de la pregunta ↓ Buscar los N chunks más similares en el vectorstore (búsqueda semántica) ↓ Construir prompt: "Pregunta: X | Contexto relevante: chunk1, chunk2, chunk3" ↓ Enviar al LLM (Azure OpenAI GPT-4o) ↓ LLM responde basándose SOLO en el contexto proporcionado
¿Por qué RAG en vez de meter todos los documentos en el contexto del LLM?
Por coste y latencia. Un contrato bancario puede ser 100 páginas. Meter 100 páginas en cada pregunta costaría una fortuna en tokens y haría cada llamada lenta. RAG selecciona solo los 3-5 fragmentos más relevantes para cada pregunta. Además, los LLMs con contextos muy largos tienen peor rendimiento en los fragmentos del "medio" del contexto (lost-in-the-middle problem). RAG mantiene el contexto pequeño y relevante.
¿Qué es el chunking y por qué importa?
Chunking es cómo divides los documentos antes de indexarlos. Si el chunk es demasiado pequeño, pierde contexto (un párrafo suelto sin saber de qué trata la sección). Si es demasiado grande, el embedding es menos preciso y el retrieval peor. El tamaño típico es 300-1000 tokens con overlap (solapamiento) del 10-20% para no perder contexto en los cortes. Para documentos estructurados como normativa o contratos, es mejor hacer chunking por sección lógica que por número fijo de tokens.
¿Qué vectorstore recomiendas y cuándo?
Cuatro opciones principales: ChromaDB para desarrollo y prototipos (fácil, local, sin infraestructura). FAISS para velocidad máxima en memoria cuando los datos caben en RAM. Milvus para producción a escala con millones de vectores, alta disponibilidad y arquitectura distribuida. Para un cliente Azure empresarial: Azure AI Search — el vectorstore gestionado de Microsoft, integrado nativamente con Azure OpenAI, con RBAC, cumplimiento normativo y sin gestionar infraestructura adicional.

RAG avanzado — si te preguntan a fondo

¿Qué problemas tiene RAG y cómo se mitigan?
Tres problemas principales: (1) Retrieval incorrecto — si los chunks recuperados no son los adecuados, el LLM genera una respuesta basada en contexto incorrecto. Mitigación: re-ranking (usar un segundo modelo para reordenar los chunks recuperados), HyDE (generar una respuesta hipotética y buscar chunks similares a ella). (2) Alucinaciones — el LLM puede generar información que no está en los chunks. Mitigación: instrucciones explícitas de "responde solo con la información del contexto, si no está di que no lo sabes". (3) Contexto fragmentado — el chunk recuperado no tiene suficiente contexto. Mitigación: overlap entre chunks, parent-child chunking.
¿Cómo medirías el éxito de un proyecto RAG en producción?
Cuatro métricas: (1) Relevancia del retrieval — ¿los chunks recuperados son los correctos para la pregunta? Se puede evaluar con un LLM-as-judge o con un dataset de preguntas con respuestas conocidas. (2) Calidad de la respuesta — ¿la respuesta generada es correcta y está fundamentada en los chunks? RAGAS es un framework específico para esto. (3) Tasa de "no sé" — ¿cuántas veces el sistema correctamente dice que no tiene información vs. alucina? Preferimos alto "no sé" a alucinaciones. (4) Satisfacción del usuario — thumbs up/down en la UI, tiempo de sesión, si el usuario tuvo que buscar la información por otro canal.

Agentes IA

Los agentes son LLMs que pueden tomar decisiones, usar herramientas externas y ejecutar secuencias de pasos para completar una tarea. Es uno de los temas más de moda — espera preguntas sobre esto.

¿Qué es un agente de IA y en qué se diferencia de un chatbot normal?
Un chatbot normal: recibes un mensaje → generas una respuesta → fin. Un agente: recibes una tarea → el LLM decide qué herramientas usar → ejecuta acciones → observa el resultado → decide el siguiente paso → repite hasta completar la tarea. La diferencia clave es la capacidad de acción: un agente puede buscar en Google, ejecutar código, consultar una base de datos, enviar un email o llamar a una API. No solo genera texto — genera texto Y actúa.
¿Qué es el patrón ReAct?
ReAct (Reasoning + Acting) es el patrón fundamental de los agentes modernos. El LLM alterna entre: Thought (razonamiento interno — "necesito saber el precio actual de IBEX para calcular esto"), Action (usa una herramienta — "buscar precio IBEX en API"), Observation (observa el resultado — "IBEX está a 11.234"), y vuelve a Thought. Este ciclo se repite hasta llegar a la respuesta final. Es el patrón que usan LangChain Agents, AutoGPT y la mayoría de frameworks de agentes.
¿Qué riesgos tienen los agentes en producción empresarial?
Cuatro riesgos principales que un consultor debe conocer: (1) Prompt injection — un usuario malicioso puede intentar que el agente ignore sus instrucciones y ejecute acciones no autorizadas. (2) Scope creep — el agente puede encadenar acciones que van más allá de lo que el usuario pretendía. (3) Costes impredecibles — un agente que entra en bucle puede hacer decenas de llamadas API antes de detenerse. (4) Trazabilidad y auditoría — en sectores regulados (banca, seguros) es obligatorio poder explicar por qué el agente tomó cada decisión. Soluciones: guardrails estrictos, límite de pasos, aprobación humana para acciones críticas (human-in-the-loop), logging completo con LangSmith o Azure Monitor.
¿Qué frameworks de agentes conoces?
LangChain es el más extendido — tiene AgentExecutor, herramientas predefinidas y buena comunidad. LangGraph (de los mismos creadores) añade control de flujo explícito con grafos de estado, más adecuado para agentes complejos en producción. Semantic Kernel de Microsoft es el más integrado con el ecosistema Azure — plugins, planners, memoria semántica. AutoGen de Microsoft Research para sistemas multi-agente donde varios agentes colaboran. Para un cliente KPMG en Azure, Semantic Kernel + Azure AI Foundry es la combinación natural.

Preguntas de consultor — razona trade-offs

Un cliente bancario quiere implementar IA para responder preguntas de sus empleados sobre normativa interna. ¿Qué arquitectura recomiendas?
RAG sobre Azure AI Search + Azure OpenAI GPT-4o. Los documentos normativos cambian constantemente — fine-tuning sería caro de mantener. Con RAG, cuando actualicen un procedimiento, reindexas ese documento y listo. Azure OpenAI garantiza que los datos normativos internos no salen de la suscripción del banco. Añadiría: citación de fuentes obligatoria en cada respuesta, temperatura 0 para respuestas deterministas, y human-in-the-loop para preguntas sobre casos edge que el sistema marque como baja confianza.
¿Cuándo NO recomendarías usar un LLM?
Cuando la respuesta correcta es única y determinista: cálculos financieros exactos, validaciones regulatorias con respuesta binaria sí/no, transformaciones de datos estructurados. Un LLM puede alucinar el resultado de una suma. Para eso, código normal. También cuando el coste no está justificado: clasificar emails en 5 categorías se puede hacer con un modelo de clasificación clásico a 1/100 del coste de un LLM. Los LLMs brillan donde el lenguaje natural, la ambigüedad y la generación de texto son el core del problema.
Un cliente tiene miedo de que sus datos acaben en modelos de OpenAI. ¿Cómo lo resuelves?
Azure OpenAI Service. Microsoft garantiza contractualmente que los datos enviados no se usan para entrenar modelos, no son accesibles para revisión humana y no salen de la región Azure configurada. Además, la comunicación puede ir por Private Endpoints — el tráfico nunca sale a internet público. Esto es lo que permite a bancos, aseguradoras y administraciones públicas españolas usar GPT-4o con cumplimiento del RGPD.

Preguntas Trampa — Las que más penalizan

Estas son respuestas que suenan bien pero que un entrevistador técnico de KPMG sabrá que son incorrectas o superficiales.

Trampa 1: "Los LLMs razonan"

Error: decir que los LLMs "razonan", "entienden" o "piensan".

Correcto: Los LLMs predicen el siguiente token basándose en patrones estadísticos. Lo que parece razonamiento es interpolación sofisticada. Implicaciones reales: no son fiables para lógica formal o matemáticas exactas, pueden contradecirse, y su "conocimiento" tiene fecha de corte.

Trampa 2: "RAG elimina las alucinaciones"

Error: decir que RAG resuelve el problema de alucinaciones.

Correcto: RAG las reduce significativamente al anclar las respuestas en documentos reales. Pero si el retrieval falla (devuelve chunks irrelevantes), el LLM puede alucinar igualmente. Y el LLM puede "confabular" combinando información de distintos chunks de forma incorrecta. RAG reduce el riesgo, no lo elimina.

Trampa 3: "Fine-tuning siempre da mejores resultados"

Error: asumir que fine-tuning > RAG en calidad.

Correcto: Para conocimiento factual actualizable, RAG supera al fine-tuning. El fine-tuning enseña estilo y formato, no hechos. Un modelo fine-tuneado con documentos de enero puede alucinar hechos de marzo. RAG con esos mismos documentos actualizados responde correctamente.

Trampa 4: "Más contexto siempre es mejor"

Error: creer que meter más contexto en el prompt mejora las respuestas.

Correcto: Los LLMs tienen el problema "lost in the middle" — dan menos importancia a información en el centro de contextos muy largos. Contextos muy largos = más latencia y más coste. El objetivo es contexto relevante y conciso, no máximo contexto posible.

Trampa 5: Confundir Airflow con MLflow

Error: usar estos términos indistintamente o confundir sus funciones.

Correcto: Airflow = orquestación de pipelines (cuándo y en qué orden se ejecuta cada paso). MLflow = tracking de experimentos y registro de modelos (qué modelo entrené, con qué datos, qué métricas obtuvo). Son complementarios, no competidores.

Trampa 6: Vectorstore = base de datos relacional con vectores

Error: describir un vectorstore como "una base de datos que guarda vectores".

Correcto: Un vectorstore es una base de datos especializada en búsqueda de similitud por distancia (coseno, euclidiana). No hace queries SQL. El motor de búsqueda es un índice ANN (Approximate Nearest Neighbor) que permite buscar los K vectores más cercanos entre millones en milisegundos.
